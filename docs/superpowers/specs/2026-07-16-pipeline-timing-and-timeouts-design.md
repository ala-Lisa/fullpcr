# 五步分析无时限执行与分步计时设计

## 目标

取消五步分析的自动超时限制，避免大数据库仍在正常计算时被主动停止；同时在快速分析进度区持续显示总运行时间、当前步骤运行时间，并持久化统计每一步的实际耗时。后台每 10 分钟检查一次进程活动，发现连续 10 分钟无活动时在 UI 标记“疑似卡住”，由用户决定是否终止。页面刷新、点击其他控件或切换页面不得重置计时或健康状态。

## 已确认的执行策略

五个步骤均不设自动超时：

| 步骤 | 自动超时 |
|---|---:|
| 基础质控 | 无 |
| 质控汇总 | 无 |
| 特异性分析 | 无 |
| obipcr 全库模拟 PCR | 无 |
| 最终综合报告 | 无 |

GUI 不再展示五步分析的 timeout 输入项。命令构建器不传 `--timeout`，五步执行器向 `run_gui_command()` 传入 `None`，由外部工具自然运行至成功退出或报错退出。

## 状态与计时设计

`fullpcr.pipeline_jobs` 继续作为项目级后台任务状态的唯一来源。任务创建时保留现有 `started_at`；每个步骤进入 `running` 阶段时新增或更新 `step_started_at`。终态继续使用 `finished_at`。状态中新增 `step_timings` 映射，以步骤键 `s1` 至 `s5` 记录每一步的 `started_at`、`finished_at`、`elapsed_seconds` 和 `status`。

后台只在任务开始、步骤切换、每 10 分钟健康检测和任务结束时持久化 ISO 8601 UTC 时间戳，不每秒写磁盘。Streamlit 的一秒轮询 fragment 在渲染时用当前 UTC 时间减去持久化时间戳：

- 总用时：运行中使用 `now - started_at`，结束后使用 `finished_at - started_at`。
- 本步用时：运行中使用 `now - step_started_at`。
- 已完成步骤用时：直接读取 `step_timings` 中冻结的 `elapsed_seconds`。

进度文字格式为：

`已完成 2/5 步 · 特异性分析 · 总用时 06:25 · 本步 05:48`

进度条下方用紧凑的五步列表显示各步状态与耗时，例如“基础质控 00:42”“质控汇总 00:03”“特异性分析 05:48（运行中）”“obipcr 等待中”“最终报告 等待中”。任务准备阶段尚无当前步骤时，只显示总用时。终态的总用时和各步用时均不继续增长。

## 无时限执行语义

`run_gui_command()` 的 `timeout` 类型扩展为 `int | float | None`。当值为 `None` 时，`subprocess.Popen.communicate(timeout=None)` 持续等待进程自然结束，不触发自动终止。外部命令返回非零退出码时仍按现有逻辑记录 `FAIL` 并停止后续步骤。

现有进程组隔离继续保留，供服务停止和用户显式终止使用；本次不由运行时长主动发送终止信号。

取消超时不改变分析算法参数、输入输出目录、重复提交保护或结果页面行为。

## 10 分钟健康检测

真实命令启动后，执行器每秒检查一次“是否收到用户终止请求”，但只每 600 秒执行一次活动采样和卡死判定。健康检测不得限制命令的总运行时间。

每次活动采样比较当前步骤完整进程组与上次样本：

- Linux `/proc` 中该进程组所有进程的用户态和内核态累计 CPU 时间是否增加。
- 当前项目输出树中普通文件的最新修改时间或总字节数是否变化；忽略 `.fullpcr_jobs` 状态目录、符号链接和非普通文件。

任一指标发生变化都视为任务仍有活动，并更新 `last_activity_at`。两项均无变化且距上次活动达到 600 秒时，将任务标记为 `suspected_stuck=true`，但不自动终止。状态文件同时保存：

- `last_health_check_at`
- `last_activity_at`
- `suspected_stuck`
- `stuck_since`
- `current_process_group`

步骤切换时重置该步骤的活动基线和疑似卡住标记。

## UI 警告与用户终止

当 `suspected_stuck=true` 时，快速分析区显示警告：当前步骤、总运行时间、当前步骤运行时间、最后活动时间和最近检测时间，并显示“终止当前分析”按钮。按钮在其他状态下不出现。

用户点击终止后：

1. 后台收到与当前 `job_id` 绑定的终止请求，避免旧页面或旧任务误杀新进程。
2. 对当前步骤的完整进程组发送 `SIGTERM`；两秒后仍未退出则发送 `SIGKILL`。
3. 当前步骤结果和整个任务记为 `CANCELLED`，中文消息为“分析已由用户终止”。
4. 停止后续步骤，释放重复提交锁，保留此前成功结果、分步耗时和健康检测记录。

页面刷新后仍从持久化状态恢复“疑似卡住”警告。终止操作只接受当前服务进程内仍然注册且 `job_id` 一致的活动任务；无法确认归属时返回结构化失败，不直接根据状态文件中的 PID 杀进程。

## 兼容与迁移

旧任务状态可能没有 `step_started_at`、`step_timings` 或健康检测字段。界面必须兼容旧状态：缺少本步时间戳时只显示总用时；缺少 `step_timings` 时不显示分步耗时列表；缺少健康字段时按“尚未发现卡死”处理，不得报错。

旧任务结果中的 `TIMEOUT` 状态继续按原样展示。新任务不读取旧的 timeout session state 值，也不把 `--timeout` 写入五步命令。

## 测试要求

- 纯函数测试覆盖五步计划的 timeout 均为 `None`，且相关 CLI 命令不含 `--timeout`。
- `run_gui_command()` 测试覆盖 `timeout=None` 时持续等待并正常返回。
- 后台任务测试覆盖每一步开始、完成和实际耗时的持久化，步骤切换时更新时间，以及终态时间冻结。
- 活动检测测试覆盖 CPU 有变化、输出有变化、两者均无变化、未满 600 秒、达到 600 秒和步骤切换重置。
- 终止测试覆盖 `job_id` 校验、`SIGTERM` 后正常退出、升级到 `SIGKILL`、`CANCELLED` 终态、锁释放和后续步骤不执行。
- AppTest 覆盖运行中总用时、本步用时和五步耗时列表，普通 rerun 后计时不归零，疑似卡住警告和终止按钮，以及旧状态字段缺失时安全降级。
- 保留现有重复提交、外部命令失败停止后续步骤、进程组隔离和完整测试套件。

## 非目标

- 不增加暂停、恢复或断点续跑。
- 不在任务仍有活动时显示终止按钮。
- 不自动终止疑似卡住的任务。
- 不更改 MFEprimer、obipcr 或 QC 的计算参数。
- 不自动重跑刚才已经超时的任务。

## 已知风险

健康检测是基于 CPU 与输出变化的保守启发式判断，不是操作系统级死锁证明。极慢且连续 10 分钟既不消耗 CPU、也不写文件的正常任务可能被标记为“疑似卡住”，但系统不会自动终止，最终决定权保留给用户。
